Hi Junhui,

Thanks for the A733 CCU series. I have been testing it on a Radxa Cubie
A7S (A733 / SUN60IW2) as part of board bring-up, and I hit a boot hang
in clk_disable_unused() that I have now root-caused and verified on
hardware. Two gates in the main CCU appear to need CLK_IS_CRITICAL.

1) ahb-cpus (0x5c0 BIT(28))

The CPU reaches the R-CCU (0x07010000) and RTC (0x07090000) register
windows through the AHB bridge this gate controls. The gate has no
Linux consumer, so clk_disable_unused() turns it off partway through
the walk; every later read of an R-CCU or RTC register then stalls the
bus, and boot hangs right after "clk: Disabling unused clocks" unless
clk_ignore_unused is set.

The walk order makes the failure look confusing: bus-r-* gates whose
parents resolve under pll-periph0 outputs are visited (and their
registers read) successfully early in the walk, while gates under the
sys-24M subtree (bus-r-i2c2 in my traces) are visited after the ahb
subtree has been processed — by then ahb-cpus is off and the first
R-CCU register read never completes.

Verified fix: with the one-line change below, the full unused-clock walk
completes on the Cubie A7S, all R-CCU and RTC gate registers read back
normally, and boot proceeds.

-static SUNXI_CCU_GATE_HWS(ahb_cpus_clk, "ahb-cpus", ahb_hws,
-			  0x5c0, BIT(28), 0);
+/*
+ * The CPU accesses the R-CCU and RTC register windows through this
+ * AHB bridge into the CPUS domain. Disabling it makes every later
+ * R-domain register read stall the bus, so it must stay on.
+ */
+static SUNXI_CCU_GATE_HWS(ahb_cpus_clk, "ahb-cpus", ahb_hws,
+			  0x5c0, BIT(28), CLK_IS_CRITICAL);

2) ahb-store (0x5c0 BIT(24)) and the store MBUS gates

In the same verified boot, once the walk completed, the SDMMC0 probe
failed with repeating "sunxi-mmc 4020000.mmc: fatal err update clk
timeout" after the walk disabled the store-lane gates. In an A/B retest
with CLK_IS_CRITICAL added to ahb-store, mbus-store (0x5e0 BIT(30)) and
mbus-msi-lite0 (0x5e0 BIT(29)), the SDMMC0 host initializes normally
with no clk-update timeouts. The SMHC sits behind the store AHB/MBUS
fabric lane, which apparently must stay up for the host's internal
clock-update handshake. These probably also want CLK_IS_CRITICAL — or
proper consumer modelling if you have a better way to express the
fabric dependency (I have not bisected whether mbus-msi-lite0 is
strictly required or only the two store gates).

If it helps I can share the full instrumented UART traces of the
unused-clock walk (hang location with the gate at 0, and the completed
walk with CLK_IS_CRITICAL).

A related note for the A733 RTC series (happy to repeat it there): on
this board "ext-osc32k" has no DT source, so ext-osc32k-gate registers
as an orphan. The register itself reads back fine once ahb-cpus stays
up — earlier I suspected the gate did not exist, but that turned out to
be the dead CPUS bridge, not a missing register. Making the external
32k input optional (boards without the crystal simply omit the parent)
seems like the right shape.

Tested-on: Radxa Cubie A7S (A733), mainline + this series + the A733
RTC and pinctrl series, GCC aarch64 build, UART console.
